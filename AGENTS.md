# AGENTS.md

## Контекст проекта

Проект: "Морской бой" версии `1.0.0` для web и мобильных устройств с режимами против людей и против бота.

Технологии:
- `React 19.2.5`
- `Vite 8.0.8`
- `TypeScript 6.0.2`
- `PHP 8.2`
- `MySQL 8.0`
- `motion 12.38.0`
- `zustand 5.0.12`
- `Playwright 1.59.1`

## Актуальная структура проекта

- `src/` - React-клиент
- `src/api/` - клиентский `fetch`-слой, запросы комнаты, session и обёртки над PHP API
- `src/components/` - верхнеуровневые экраны и composition-слой UI
- `src/components/app/` - общие app-компоненты, toast и служебный UI
- `src/components/game/` - боевые и setup-компоненты, sidebar, модалки и игровые actions
- `src/components/lobby/` - создание комнаты, вход по коду и старт bot-room
- `src/hooks/` - пользовательские настройки, presence и orchestration уровня приложения
- `src/hooks/roomSync/` - синхронизация состояния комнаты и polling-логика
- `src/styles/` - разбитые CSS-модули темы, foundation, responsive и battle-оформления
- `api/` - PHP API
- `api/actions/` - серверные сценарии комнат, расстановки, боя, сдачи и реванша
- `api/game_*.php` - модульная серверная игровая логика, бот, правила, room и board core
- `api/i18n.php` - серверная локализация ответов и last-action сообщений
- `shared/` - общие типы, контракты комнаты и чистая игровая логика
- `shared/game/` - правила постановки, боя, авторасстановки, board/room helpers и типы домена
- `database/` - SQL-схема базы данных
- `scripts/` - release, shared/contract/cleanup/smoke/ui тесты и вспомогательные runner-скрипты
- `scripts/.tmp/` - временные артефакты тестов, не хранить в git
- `public/` - статические файлы, favicon, robots и manifest
- `release/` - staging релиза и итоговый архив для хостинга
- `server/` - служебные серверные файлы окружения, если они используются локально

## Цели продукта

- дать двум игрокам на разных устройствах играть через комнату по коду
- придерживаться классических правил морского боя
- держать сервер авторитетным для критичной игровой логики
- работать на обычном shared hosting без постоянного Node-процесса
- сохранить яркий морской интерфейс с анимациями и понятной мобильной адаптацией

## Актуальные возможности

- создание комнаты и вход по коду
- отдельный режим игры с ботом без ожидания реального второго игрока
- ручная расстановка флота с preview перед подтверждением
- авторасстановка флота без автоподтверждения
- старт матча только после готовности обоих игроков
- повторный ход после попадания или потопления
- перезапуск таймера хода после каждого успешного попадания или потопления
- сдача во время боя
- реванш по согласию обоих игроков
- мгновенный реванш с ботом без ожидания второго участника
- мобильное боковое меню и упрощённая боевая боковая панель
- настройки интерфейса: светлая/тёмная тема, язык `ru/en/uk`, стиль боевой страницы
- presence-пинг и счётчик онлайна на стороне клиента
- серверные локализованные сообщения и `lastAction`
- браузерные UI-регрессионные тесты на ключевые React-переходы и действия
- smoke-, cleanup-, contract- и UI-проверки сценариев, включая bot-room

## Правила игры

- поле `10x10`
- флот каждого игрока: `1x4`, `2x3`, `3x2`, `4x1`
- корабли ставятся только горизонтально или вертикально
- выход за границы, пересечения и касание по стороне или диагонали запрещены
- повторно стрелять в ту же клетку нельзя
- при промахе ход переходит сопернику
- при попадании или потоплении игрок ходит снова
- таймер хода стартует заново после любого нового хода, включая попадание и потопление
- тип корабля раскрывается только при потоплении
- победа наступает при уничтожении всего флота соперника

## Правила для ИИ-агентов

- сначала читать `shared/game/`, затем `api/index.php`, затем клиентские компоненты
- после `api/index.php` смотреть `api/action_handlers.php`, `api/actions/` и соответствующие `api/game_*.php`
- не ломать сервер-авторитетную модель
- не переносить проверку правил только в UI
- не раскрывать координаты чужих кораблей в API-ответах
- при клиентской работе различать серверный `RoomState` и клиентский `SanitizedRoomState`
- режим с ботом держать в том же серверном контракте комнаты, а не в отдельной клиентской симуляции
- при изменении API синхронно обновлять PHP-обработчики, клиентский `fetch`-слой и типы в `shared/`
- при изменении игровых правил синхронно обновлять этот файл, `README.md` и связанные пользовательские тексты
- при изменении локализации синхронно обновлять клиентский `src/i18n.ts` и серверный `api/i18n.php`
- при изменении настроек интерфейса учитывать `theme`, `locale`, `battleGraphicsStyle` и не ломать сохранение в session/localStorage
- временные тестовые файлы оставлять в `scripts/.tmp/` и не коммитить
- при изменении постановки кораблей сохранять ручную расстановку и наглядный preview перед подтверждением
- во время боя не возвращать в sidebar старые блоки таймера, статусов игроков и служебной информации без явной причины
- сохранять единый визуальный стиль: насыщенная палитра, морская тема, выразительная типографика, glow-эффекты
- не добавлять новые библиотеки без явной причины
- после любых изменений сразу выполнять `npm run release` и оставлять актуальный архив в `release/`

## Кодовые соглашения

- использовать TypeScript `strict`
- общие типы и игровые контракты держать в `shared/`
- предпочитать чистые функции для игровых расчётов
- UI-компоненты делать небольшими и специализированными
- не допускать расхождения правил между `shared/game.ts`, `shared/game/` и PHP-обработчиками
- в клиенте держать API-запросы внутри `src/api/`, а не размазывать `fetch` по компонентам
- в серверном PHP поддерживать модульное разбиение по ответственности, а не раздувать `api/index.php`
- при изменении тестовых сценариев обновлять соответствующие скрипты в `scripts/`
- для критичных переходов React держать `data-testid`, если они уже используются UI-тестами

## Что проверять после изменений

- создаётся ли комната
- создаётся ли отдельная комната с ботом
- может ли второй игрок подключиться по коду
- проходит ли постановка кораблей без нелегальных раскладок
- работает ли авторасстановка и остаётся ли подтверждение ручным шагом
- стартует ли игра только после готовности обоих игроков
- правильно ли меняется ход после промаха, попадания и потопления
- перезапускается ли таймер после попадания и потопления
- не раскрываются ли чужие корабли раньше времени
- показываются ли в боевом sidebar только актуальные элементы боя
- завершается ли матч корректной победой или сдачей
- добивает ли бот найденный корабль после попадания
- работают ли локализация, переключение темы и настройки графики боя
- обновляется ли presence без влияния на игровой polling
- проходят ли `npm test` и `npm run release`
